# Spec: Internacionalização do site Satriz (PT · EN · ES)

**Data:** 2026-07-03
**Status:** aprovado pelo Asafe (design verbal); aguardando revisão da spec escrita

## Objetivo

Atrair clientes internacionais: o site público deve existir em português (padrão), inglês e espanhol, com URLs indexáveis pelo Google e um seletor de idioma na nav. O painel `/admin` e as APIs ficam fora do escopo (permanecem em PT).

## Decisões de produto (fechadas com o usuário)

| Decisão | Escolha |
|---|---|
| Objetivo | SEO / captação internacional (não só conveniência) |
| Escopo | Só o site público; `/admin` e `/api` intocados |
| Primeira visita | Auto-detectar idioma do navegador e redirecionar |
| Escolha manual | Persiste em cookie e vence a auto-detecção |
| Abordagem | **A — biblioteca `next-intl`** (rotas `[locale]`, mensagens JSON) |
| Slugs | Iguais nos 3 idiomas (`/en/servicos`, não `/en/services`) |
| Traduções | Geradas pelo Claude; revisão posterior do Asafe |

## Arquitetura

### URLs
- PT (padrão): `satriz.club/`, `/servicos`, `/projetos`, `/contato`, `/criador` — **sem prefixo** (`localePrefix: "as-needed"`), URLs atuais preservadas.
- EN: `satriz.club/en/...` · ES: `satriz.club/es/...`
- `hreflang` alternates em todas as páginas públicas (next-intl/metadata `alternates.languages`).

### Estrutura de pastas
```
app/
  [locale]/            ← NOVO: páginas públicas movem pra cá
    layout.tsx         ← NextIntlClientProvider + metadata por idioma + fontes
    page.tsx           ← home (Scene3D/Hero/ProofBar)
    servicos/page.tsx
    projetos/page.tsx
    contato/page.tsx
    criador/page.tsx
  admin/               ← fica onde está (PT, protegido)
  api/                 ← fica onde está
messages/
  pt.json  en.json  es.json   ← TODO o texto visível do site público
i18n/
  routing.ts           ← locales: ["pt","en","es"], defaultLocale: "pt", localePrefix: "as-needed"
  request.ts           ← carrega messages do locale ativo
  navigation.ts        ← Link/useRouter/usePathname locale-aware (wrappers next-intl)
```

### Middleware (ponto delicado)
Um único `middleware.ts` com duas responsabilidades, nesta ordem:
1. **Auth do admin (lógica atual, inalterada):** `/admin*` e `/api/leads*` → checa cookie, redireciona/401.
2. **i18n (next-intl):** todas as outras rotas → detecção Accept-Language na primeira visita, cookie `NEXT_LOCALE` para escolha manual, redirect/rewrite de locale.

Matcher exclui `/api`, `/_next`, arquivos estáticos; `/admin` passa só pela etapa 1.

### Seletor de idioma na Nav
- Desktop: pílula de vidro (mesmo estilo `glass` atual) com `PT · EN · ES`; idioma ativo em âmbar (`text-flush`); fica entre a pílula de links e o CTA.
- Mobile: linha com os 3 idiomas dentro do menu hambúrguer.
- Trocar idioma mantém a rota atual (`/projetos` → `/en/projetos`) usando `useRouter`/`usePathname` do `i18n/navigation.ts`, disparando a transição de cortina existente; o next-intl grava o cookie.

### Migração de conteúdo
- `lib/data.ts` (208 linhas: NAV_LINKS, ESTATUTO, DIVISOES, PROJETOS, PAPEIS, SERVICOS, DIFERENCIAIS, PROCESSO, FAQ...) → arrays/objetos equivalentes nos `messages/*.json`; componentes leem via `useTranslations()`/`t.raw()`. Constantes não-textuais (WHATSAPP_HREF base, SOCIALS, STACK, hrefs) permanecem em `lib/data.ts`.
- ~18 componentes/páginas com texto hardcoded migram pra `useTranslations` (client) ou `getTranslations` (server): Hero, Scene3D (manifesto, serviços, etapas, matriz, painel contato), EntryGate, Nav, Footer, WhatsAppFab (incl. texto pré-pronto do wa.me), Orcamento, CriadorForm, Contato, Club, ClubMini, Estatuto, PorQue, Processo, ProofBar/Trabalho/Servicos/Divisoes/Faq/Credential conforme tenham texto.
- Mensagens de erro/sucesso dos formulários: as exibidas no client vêm das messages (as respostas da API podem continuar em PT; o client mostra a sua própria mensagem traduzida).
- Metadata por locale: `generateMetadata` no `app/[locale]/layout.tsx` (title, description, OG/Twitter) + `lang` dinâmico no `<html>`.

### PageTransition
- Passa a usar `useRouter`/`usePathname` de `i18n/navigation.ts` (locale-aware) no lugar dos de `next/navigation`. Comportamento (cortina cobre → rota troca → espera montar → revela) inalterado.

## O que NÃO muda
Admin, APIs, modelo de dados/leads, cena 3D em si, vídeos, imagens, design system, EntryGate (só textos), transição de cortina (só a origem do router).

## Casos de borda
- `/admin` nunca recebe prefixo de locale (excluído do matcher i18n).
- Link do footer/rodapé "desenvolvido por Satriz Club" em sites de clientes aponta pra raiz → cai na detecção normal.
- Usuário PT que escolheu EN: cookie vence; voltar pra PT pelo botão regrava o cookie.
- 404: `app/[locale]/not-found.tsx` traduzido (bônus barato; se complicar, fica PT).

## Verificação (critérios de aceite)
1. `npx tsc --noEmit` e `next build` limpos.
2. `curl -H "Accept-Language: en"` na raiz → redirect `/en`; `es` → `/es`; `pt` → fica na raiz.
3. `curl /en/servicos` e `/es/servicos` → 200 com conteúdo traduzido; `/servicos` → PT.
4. `/admin` continua protegido (307 → login sem cookie) e sem prefixo.
5. HTML contém `hreflang` para os 3 idiomas e `<html lang>` correto.
6. Botão na nav troca idioma mantendo a página, com transição de cortina, e a escolha persiste após fechar/abrir o navegador.
7. Formulários de lead continuam gravando no Mongo nos 3 idiomas.

## Riscos
- **Mover páginas quebra imports** → mitigado por `tsc` + smoke test de todas as rotas.
- **Middleware combinado** → testar auth do admin ANTES e DEPOIS da mudança (mesmos curls da sessão anterior).
- **Volume de strings** → migração em etapas por componente, verificando a cada lote.
- **Qualidade das traduções** → Asafe revisa `en.json`/`es.json` depois; marcar com comentário no PR.
